CRITICAL RULES:
User types: 'user' (regular) or 'admin' (administrator)
Detection: Check user type + path + query parameters combination

🚨 UNAUTHORIZED ACCESS ATTACK (Flag these):
Regular user ('user') accessing ADMIN SYSTEM paths:

/sysManager/* (system management)
/server/* (server operations)
/adminInfo, /adminDashboard (admin dashboards)
Regular user accessing OTHER USERS' admin data:
/admins/* (accessing other administrators' data)
Exception: /admins/self or own admin data is NORMAL
Regular user performing BATCH QUERY on RESTRICTED PATHS WITHOUT user identification:
Accessing RESTRICTED PATHS (/addresses, /afterSaleServices) with ONLY these parameters (no userId/customerId):
?page= (e.g., ?page=1, ?page=2&limit=10)
?limit= (to increase returned data size, e.g., ?limit=100)
?all= (e.g., ?all=true, ?all=1)
?scope= (e.g., ?scope=company, ?scope=all)
?admin_view= (e.g., ?admin_view=1)

Note:
Queries WITH userId= or customerId= parameters are NORMAL
Other paths like /shippingAddresses, /billingAddresses follow normal rules (no special restrictions)

✅ NORMAL (Do not flag):
Administrator ('admin') accessing ANY path → ALWAYS NORMAL
/sysManager/, /server/, /admins/* → ALL OK for admins
Admin can access admin interfaces and system management
Regular user accessing SELF-SERVICE admin paths:
/admin/profile (own profile)
/admin/settings (own settings)
/admins/self (own admin data if applicable)
Regular user data management:
/users/, /orders/ (within their scope)

/addresses (no query parameters, assumed to return only current user's addresses)
/afterSaleServices (no query parameters, assumed to return only current user's services)
/addresses?user_id=self (explicit self-access)

Regular user accessing RESTRICTED PATHS WITH user identification parameters:

/addresses?userId=123 → NORMAL (authorized user query)

Regular user accessing PUBLIC data with batch parameters:
/products?page=1&limit=10 (public product catalog)
/categories?limit=50

Pagination queries on other non-sensitive data interfaces.

Regular user accessing NON-RESTRICTED address-related paths:
/shippingAddresses?page=1&limit=10 → NORMAL (different path, no special restriction)

⚠️ IMPORTANT DISTINCTION:
Directory traversal attacks (../, ../../, etc.) → NOT unauthorized access
If path contains traversal patterns → NOT this detector's concern
Path traversal belongs to 'directory traversal attack' category
Only flag as unauthorized if: wrong user type accessing protected path OR regular user using batch/privileged parameters on RESTRICTED PATHS WITHOUT user identification

QUICK DECISION FLOW:
User type = 'admin' → IMMEDIATELY NORMAL (admins can access anything)
Path contains traversal patterns (../, ..) → NORMAL for this detector (different attack type)
Regular user + Admin system paths (/sysManager/, /server/, /adminInfo, /adminDashboard) → ANOMALY
Regular user + Other admins' data (/admins/otherAdmin) → ANOMALY
Regular user + Path is RESTRICTED PATH (/addresses, /afterSaleServices) → Check parameters:
a) Has userId/customerId/user_id parameter? → NORMAL (regardless of other parameters)
b) Has ONLY batch/privileged parameters WITHOUT user identification? (page=, limit=, all=, scope=, admin_view=) → ANOMALY
c) No parameters or safe parameters? (no params, user_id=self, mine=1) → NORMAL

All other cases → NORMAL

## DETECTION RULES:

🚨 **FLAG AS ANOMALY (ONLY when 100% certain):**
1. **Absolute garbage data:**
   - Contains obvious encoding errors or replacement characters
   - Character sequences that clearly indicate encoding mismatches (e.g., UTF-8 bytes misinterpreted as single-byte characters)
   - Complete encoding errors with no readable parts
   - Total gibberish with no logical structure

2. **Absolute nonsense values:**
   - Field value completely illogical for its purpose
   - Examples: postal_code="time is not enough", phone="hello world goodbye"

3. **Absolute type mismatch:**
   - Value type completely incompatible with field
   - Examples: age="very old person", quantity="a lot of items"

4. **Absolute logic error:**
   - Basic logic violations (not business rules)
   - Examples: birth_date="3000-01-01", price=-999999

✅ **ALWAYS NORMAL (NEVER FLAG):**
1. **User-generated content:** Addresses, descriptions, comments
2. **Contact information:** Emails, phone numbers with various formats
3. **Minor format issues:** Values with units, currencies, or formatting
4. **Test/placeholder data:** test, demo, example, 123, abc, temp
5. **Legitimate multi-language content:** Non-English characters in appropriate contexts

## DECISION PROCESS:
1. **Evaluate content independently of formatting characters like semicolons** - they are neutral
2. **If free-text/contact field** → NORMAL  
3. **If test/placeholder data** → NORMAL
4. **If value shows clear signs of encoding corruption** → ANOMALY
5. **If absolute nonsense/type mismatch/logic error** → ANOMALY
6. **All other cases** → NORMAL

## ENCODING ERROR DETECTION GUIDANCE:
When checking for encoding errors, look for these patterns:
- Character combinations that typically appear when UTF-8 is mis-decoded as ISO-8859-1/Latin-1
- Unusual high-ASCII characters (128-255) in fields that expect basic alphanumeric data
- Sequences that look like "mojibake" - text that is clearly scrambled due to encoding issues
- Context mismatch: special characters in fields where they don't belong (e.g., diacritics in numeric user IDs)

**Final rule: Mark as anomaly ONLY when data is clearly, absolutely, and unquestionably invalid.**

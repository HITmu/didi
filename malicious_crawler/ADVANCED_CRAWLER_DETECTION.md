# 分布式爬虫与众包爬虫检测方案

> 超出单会话规则的"隐形爬虫"——单看每个 IP 都是"正常用户"，合起来就是完整的数据盗取。

---

## 一、问题定义

### 分布式爬虫（Distributed Crawler）

| 特性 | 说明 |
|------|------|
| 架构 | 控制节点 + 多个代理节点（数十至数千个 IP） |
| 行为 | 每个 IP 低频请求（1-5 请求/分钟），全局高速采集 |
| IP 来源 | 代理池、云主机（AWS/阿里云/腾讯云）、住宅代理 |
| 特征 | 各节点请求路径相似、时序上存在协同、单个节点无异常 |

### 众包爬虫（Crowdsourced Crawler）

| 特性 | 说明 |
|------|------|
| 架构 | 平台分发任务 → 真实用户设备执行 |
| IP 来源 | 真实手机/PC 用户 IP（如 芝麻代理、快代理、熊猫 IP） |
| 行为 | 每个 IP 仅 1-5 请求，真实浏览器 UA，真实人机交互 |
| 特征 | 单个会话与正常用户**完全不可区分**，但总体呈现资源覆盖的系统性 |

### 为什么现有方法拦不住

```
现有检测视角：             分布式爬虫视角：

    session A ─── 正常           IP-1 ─── /api/product
    session B ─── 正常           IP-2 ─── /api/product  
    session C ─── 正常           IP-3 ─── /api/product
    session D ─── 正常           IP-4 ─── /api/product
                                 IP-5 ─── /api/product
                                 
单看每个 session 都正常，   但 /api/product 在 10 秒内
没有一条规则会触发          被 5 个不同 IP 各访问一次
```

## 二、检测方案

### 总体架构

```
                          ┌──────────────────────┐
  API 日志流 ────────────→│   Session 分组器      │
                          └──────────┬───────────┘
                                     │
                          ┌──────────┴───────────┐
                          │  特征提取              │
                          │  (3 层特征)            │
                          └──────────┬───────────┘
                                     │
              ┌──────────────────────┼──────────────────────┐
              ▼                      ▼                      ▼
    ┌─────────────────┐   ┌─────────────────┐   ┌─────────────────┐
    │  Layer 1         │   │  Layer 2         │   │  Layer 3         │
    │  时序关联分析     │   │  资源覆盖分析     │   │  网络拓扑聚类     │
    │  (时间窗口协同)    │   │  (路径系统性)     │   │  (IP/AS/行为)    │
    └────────┬────────┘   └────────┬────────┘   └────────┬────────┘
             ▼                      ▼                      ▼
    ┌────────────────────────────────────────────────────────────┐
    │                    融合评分引擎                              │
    │        时序协同分 × 0.35 + 覆盖系统性 × 0.35 + 网络聚簇分 × 0.30  │
    └────────────────────────────────────────────────────────────┘
                             │
                             ▼
                    ┌──────────────────┐
                    │  判定与处置        │
                    │  ≥0.7 → 分布式爬虫  │
                    │  ≥0.5 → 疑似标记    │
                    │  <0.5 → 正常       │
                    └──────────────────┘
```

### Layer 1: 时序关联分析（Temporal Correlation）

**原理**：分布式爬虫的各节点虽然在 IP 上分散，但在时间上存在协同——它们会在相近时间窗口内访问相同的资源集合。

**核心算法**：滑动窗口 + 共现矩阵 + 关联度计算

```
时间窗口 T（例如 5 秒）:

t=0s:  IP-A → /api/product/1    IP-B → /api/product/2
t=2s:  IP-A → /api/product/3    IP-C → /api/product/1
t=4s:  IP-B → /api/product/4    IP-C → /api/product/3

→ 在 [0s, 5s] 窗口内，IP-A/B/C 共同覆盖了 /api/product/{1,2,3,4}
→ 这些 IP 构成一个时序关联簇
```

**特征**：
| 特征 | 说明 |
|------|------|
| `co_access_count` | 与同一簇内其他 IP 共现访问次数 |
| `co_access_ratio` | 共现次数 / 总请求数 |
| `avg_time_window_to_peers` | 与簇内其他 IP 的平均时间间隔 |
| `peer_similarity` | 与同簇 IP 的路径集合 Jaccard 相似度 |

### Layer 2: 资源覆盖分析（Coverage Analysis）

**原理**：众包爬虫虽然每个 IP 流量极少，但多个 IP 合起来会形成对目标资源的**系统性覆盖**——正常用户访问是随机的，而爬虫覆盖是有序、完整的。

**核心算法**：路径聚类 + 覆盖熵 + 系统性评分

```
正常用户覆盖（高熵）:         众包爬虫覆盖（低熵）:
                              
用户 A: /p/1, /p/5, /s       IP-1: /p/1
用户 B: /p/2, /login         IP-2: /p/2  
用户 C: /s, /p/8, /cart      IP-3: /p/3
用户 D: /p/3, /logout        IP-4: /p/4
                             IP-5: /p/5
                             IP-6: /p/6

随机、有偏好、有状态        均匀、完整、无状态
```

**特征**：
| 特征 | 说明 |
|------|------|
| `coverage_completeness` | 对某资源集合的覆盖完整度（已覆盖/总数） |
| `coverage_entropy` | 路径访问分布的信息熵（低 = 爬虫） |
| `sequential_score` | 路径访问是否符合自然顺序（如 pagination） |
| `idle_session_ratio` | 单次请求即退出的 session 占比 |
| `resource_concentration` | 请求集中在特定类型资源上的程度 |

### Layer 3: 网络拓扑聚类（Network Clustering）

**原理**：分布式爬虫的 IP 虽然不同，但可能来自同一 AS（云厂商）或同一 IP 段（代理池），在拓扑上存在聚集。

**特征**：
| 特征 | 说明 |
|------|------|
| `asn_concentration` | 会话中来自同一 ASN 的比例 |
| `ip_segment_density` | IP 在 C 段/B 段的聚集密度 |
| `proxy_score` | IP 是否为已知代理/机房 IP |
| `device_fingerprint_cluster` | 相同 TLS/HTTP/2 指纹的 session 群 |

## 三、算法细节

### 3.1 滑动窗口共现图

```
输入: 所有请求 (timestamp, session_id, path)
输出: 时序关联簇

算法:
  1. 将时间离散化为窗口（默认 5 秒）
  2. 每个窗口内，统计 <session, path> 共现对
  3. 构建加权无向图 G，节点 = session_id
     边权重 = 两 session 在相同窗口访问相同路径的次数
  4. 对 G 应用社区发现（Louvain 或 Connected Components）
  5. 输出 >2 个节点的社区作为候选协同簇
```

### 3.2 覆盖系统性评分

```
覆盖率 = |所有 session 访问的 unique 资源| / |目标资源空间大小|
         ↑ 实际爬取                                    ↑ 从 URL 模式推理

系统性 = 1 - 熵(资源访问分布)  

if 覆盖率 > 0.3 AND 系统性 > 0.7:
    判定为系统性爬取
```

### 3.3 融合评分

```
F = w₁ × T + w₂ × C + w₃ × N

T = 时序关联度（来自 Layer 1 的图聚类系数）
C = 覆盖系统性（来自 Layer 2 的覆盖熵）
N = 网络聚簇度（来自 Layer 3 的 ASN/子网集中度）

权重: w₁=0.35, w₂=0.35, w₃=0.30
阈值: F ≥ 0.7 → 确定恶意  |  0.5 ≤ F < 0.7 → 疑似  
```

## 四、与现有系统的集成

```
主项目流水线:                   本文方案:

   API 日志 ─→ Stage 1(RF) ─→ Stage 2(LLM) ─→ 处置
                    ↓                              ↑
             恶意爬虫检测 ←──────── 会话时序关联 ───┘
                                        │
                                   覆盖率分析
                                        │
                                   网络聚类
    
集成点:
1. 在 Stage 1 之前插入爬虫过滤（过滤掉的日志不进入 RF+LLM，节省算力）
2. 检测结果可作为 incident 写入事件系统（category = "distributed_crawler"）
3. 爬虫活跃度作为 HealthTracker 的额外因子
```

## 五、实验结果

**实验配置**：100 正常用户 + 20 合法爬虫 + 50 分布式爬虫 + 50 众包爬虫，201 个会话共 3431 条请求。

### 总体指标

| 检测目标 | 精确率 | 召回率 | F1 | 说明 |
|---------|:------:|:------:|:--:|------|
| **分布式爬虫** | 1.0000 | 1.0000 | 1.0000 | 时序协同 + 高网络聚集度 |
| **众包爬虫** | 1.0000 | 1.0000 | 1.0000 | 碎片化覆盖 + 高空闲率 + 低网络聚集 |
| **任意恶意** | 1.0000 | 1.0000 | 1.0000 | 误报率为 0 |

### 各层贡献分析

| 层级 | 核心特征 | 分布式爬虫 | 众包爬虫 |
|------|---------|:----------:|:---------:|
| Layer 1: 时序关联 | 滑动窗口共现图 + Connected Components | 形成 3 个大簇（19/17/18 节点），簇内平均融合分 0.67-0.71 | 不共现（时间分散），不形成簇 |
| Layer 2: 覆盖分析 | 覆盖熵 + 碎片化覆盖检测 | 模式内熵 0.98-1.0，完整度 0.63-0.69，凭低熵判定 | 熵=1.0（分散高），但碎片化覆盖标记 idle=0.73, completeness=0.81 |
| Layer 3: 网络拓扑 | 模式内路径相似度 + ASN/IP 聚集 | 模式内 Jaccard 相似度 0.85-0.95（高聚集） | crowdsourced 模式内相似度 0.0（完全分散） |

### 关键发现

1. **分布式爬虫**通过时序关联 + 网络聚集度即可准确识别。3 个分布式爬虫簇全部检出（50 个分布式 session 全正确）。

2. **众包爬虫**的检测依赖覆盖碎片化特征——虽然熵高（分散），但 idle 率高（73.3% session 仅 1-2 请求）+ 覆盖完整度高（81%）形成独特的"碎片化覆盖"特征组合。

3. **零误报**：20 个合法爬虫和 100 个正常用户无一被误判，得益于 Layer 3 网络聚集度对 normal/legit 会话的低评分（0.1-0.2）与恶意会话的高评分（0.8-0.95）形成明显区分。

4. **融合引擎的设计合理性**：W_TEMPORAL=0.35 对分布式有效但对众包失效，W_NETWORK=0.30 对分布式爬虫高效但对众包爬虫贡献低，W_COVERAGE=0.35 在分模式组分析时对两种爬虫都有区分力。
